# ТЗ и план работ (приоритет): автонастройка + каскады, затем MTProto

Дата: 2026-04-07

## 1) Приоритеты разработки

## P0 (сразу)
Стабилизировать:
1. автонастройку нод (`Xray` и `Hysteria`);
2. корректный деплой и работу каскада `Xray -> Hysteria -> Xray`.

## P1 (после P0)
MTProto/MTProxy:
1. автовыдача для пользователя;
2. привязка к подписке;
3. базовая статистика в карточке подписки.

## P2+ (дальше)
Остальные блоки roadmap:
- Next.js migration,
- Sing-box full runtime,
- failover/policy engine,
- Prometheus advanced,
- WireGuard underlay и т.д.

## 2) Область P0

## 2.1 Что должно работать

1. Автонастройка `Xray`:
- установка бинаря,
- генерация ключей/конфига,
- запуск сервиса,
- корректная диагностика ошибок.

2. Автонастройка `Hysteria`:
- установка бинаря,
- TLS/конфиг,
- перезапуск сервиса,
- sidecar runtime (если гибрид включен).

3. Каскад `Xray -> Hysteria -> Xray`:
- deploy без ручных правок на серверах,
- overlay на Hysteria-хопе поднимается автоматически,
- линк становится `online` и сохраняет состояние после sync.

## 2.2 Что не входит в P0

- Полная миграция UI на Next.js.
- Новые протоколы кроме тех, что нужны для рабочего гибридного каскада.
- Полноценная per-user статистика MTProxy.

## 5) Технические задачи P0

## 5.1 Автонастройка: hardening

1. Preflight-проверки перед setup:
- SSH auth,
- доступность внешних источников загрузки,
- свободные порты,
- базовые зависимости.

2. Единый retry/fallback policy для загрузок:
- installer script,
- релизные архивы,
- mirror fallback.

3. Устойчивый service activation check:
- после restart проверка состояния с retry-окном (`activating` -> `active`),
- корректная классификация ошибок (`port_in_use`, `download_failed`, `auth_failed`, `service_failed`).

4. Защита от портовых конфликтов:
- детектор “панель и xray на одном хосте с портом 443”,
- fail-fast с понятным сообщением и рекомендацией порта.

## 5.2 Каскады: X-H-X

1. Деплой sidecar на Hysteria-хопе:
- идемпотентно,
- с валидацией конфига и порта,
- с health-check после старта.

2. Overlay-применение в hysteria config:
- marker presence,
- сохранение после sync/redeploy.

3. Автоматический post-deploy smoke-check:
- сервисы active,
- sidecar слушает SOCKS порт,
- overlay marker на месте,
- xray binary присутствует.

## 5.3 API/панельная диагностика

1. Setup Status API: стабильное возвращение `state` и `error`.
2. Логи в UI: выделение root-cause.
3. Чек-лист результата setup/cascade deploy прямо в панели.

---

## 6) Acceptance Criteria P0

1. На стенде проходит:
- автонастройка `Xray` (3/3 подряд),
- автонастройка `Hysteria` (3/3 подряд).

2. Каскад `Xray -> Hysteria -> Xray`:
- deploy успешен,
- статус линка `online`,
- после `sync` остается рабочим.

3. На негативных кейсах:
- занятый порт,
- битая загрузка,
- неверный SSH

система возвращает корректный нормализованный код ошибки и понятный текст.

4. Smoke-check для гибридного хопа проходит автоматически и вручную скриптом.

## 7) План выполнения P0 (итерации)

## Итерация 1 — setup reliability
1. Preflight + port conflict guard.
2. Service activation retry window.
3. Нормализация error taxonomy.

Результат:
- setup не падает “немо”,
- причины понятны в UI.

## Итерация 2 — cascade reliability
1. Idempotent sidecar deploy.
2. Overlay persistence on sync.
3. Авто smoke-check после deploy.

Результат:
- стабильный `X-H-X` deploy.

## Итерация 3 — e2e stabilization
1. Прогон матрицы на 3 серверах.
2. Фиксы регрессий.
3. Freeze критериев P0 и handover.

Результат:
- подтверждённая рабочая связка.

## Итерация 3.5 — clean rerun gate
1. Полностью чистые серверы / свежие снапшоты.
2. Повторная установка панели и создание админа.
3. Повторная автонастройка `Xray`, `Hysteria` и полной цепочки.
4. Повторный импорт подписки в реальном клиенте (`Happ`) без использования старой подписки.

Результат:
- `P0-release milestone` считается закрытым только после успешного clean rerun.

## 9) Fallback с Sing-box (если нужно)

Если в процессе P0 не удаётся добиться стабильного гибридного runtime на Xray-sidecar:
1. включаем технический fallback: `sing-box sidecar` на Hysteria-хопе;
2. сохраняем тот же UX в панели;
3. включаем через feature-flag и прогоняем ту же acceptance матрицу.

Решение о включении fallback принимается только после итогов Итерации 2.

## 10) Следующий этап после P0

Если clean rerun успешен:
1. фиксируем P0 как завершенный;
2. открываем `Sing-box PoC` под feature-flag;
3. используем текущую рабочую цепочку как baseline для сравнения с Sing-box runtime;
4. не переносим всю систему на Sing-box сразу, пока не пройдены критерии `PoC -> Beta -> Migration`.
